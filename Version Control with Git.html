What is version control?

Version control is a way of recording changes to files over time so that we can revert back to specific versions later.

Manual version control

The most common version control used would be versioned file names like myFile_2019.04.27.txt or myFile_v1.0.txt. This approach gets the job done, but it takes up a big storage space and have little information on how the versions of the files differ.

http://www.phdcomics.com/comics/archive.php?comicid=1531

Automated version control via Git

Git is the most commonly used version control system today and is quickly becoming the standard for version control. Git is a distributed version control system. While git can track changes of any file type it is most useful to track the text-based source files such as .txt, .Rmd, .R, etc.

Git saves the entire history of changes in the project

- commented, time-stamped, author identified

Benefits of version control:

  • easy to see what changed when, why, and by whom

  • jump back to any point in time

  • freedom to experiment

    • try out a crazy idea
    • can always go back to the last known good version
    • keep working on both in parallel

  • manage contributions from multiple people

Why use Git and not fill in the blank?

  • it is very fast, even for projects with hundreds of files and years of history

  • doesn’t require access to a server

  • extremely popular, no strong reason to pick anything else (unless required)

Git on the local computer

All Git functions are accessed from the command line with git followed by the function name, for example to initialize git the command is git init. The commands used in this section are summarized in the following infographic. Please refer to it as we progress through the examples.

Initialize a repo (git init)

First, create a directory on your computer named TestRepo. Then, open your terminal/GitBash, navigate to the TestRepo directory, and initialize it as a repo with

git init

You should see an output similar to the following showing that Git knows this directory is a repo.

Initialized empty Git repository in C:/LocalgitRepos/TestRepo/.git/

If you now list all the files in the TestRepo directory with ls -a, you will see a .git directory that keeps the version control of the directory TestRepo. Now you can add files to the directory TestRepo as usual, the .git directory will track changes of each file in the directory TestRepo.

Add files (git add)

Create a README.md file using a text editor of your choice and include the following as the contents of this file.

This is a README for my first Git repo.

Now that we have added a new file, we can query if there are any differences between our local TestRepo directory and the version controlled repo with

git status

The result from git status is telling us that the file README.md is in our local directory TestRepo that is not in the version controlled repo:

$ git status
On branch master

Initial commit

Untracked files:
  (use "git add <file>..." to include in what will be committed)

        README.md

nothing added to commit but untracked files present (use "git add" to track)

This means that README.md file is untracked by git. The last line of the git status output instructs us to use git add to add the file README.md to the staging area of the version controlled repo.

We can either add just this file by specifying the file name:

git add README.md

Or if we have more than 1 file to add at once, we could add all with either of these commands (this also works with 1 file if you are lazy to type a single file name):

git add .
git add --all

If we ask again for the git status, it says that the file README.md has been staged but it is not committed to the local versoned repo:

$ git status
On branch master

Initial commit

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)

        new file:   README.md

Git staging area (index)

If the file is tracked but not committed (i.e. version saved), where is it? Git has a staging area (also called the index) where a version of a file is stored prior to it being more permanently saved as a committed version.

This is useful because sometimes you may want to:

  • save a version as a back-up in the short-term but do not want to permanently save that version in the long-term
  • complete many small changes separately but save them all together as 1 new version
  • compare your current version with the last committed version without causing any conflicts in your repo

Commit changes (git commit)

Once you have a version you want to save permanently, you need to commit it from the index (staging area) to the local repo.

To save the changes we need to commit using git commit -m 'message about this version':

git commit -m 'add my README.md file'

Note: Best practice is for git committo provide a message -m.

Result from the git commit:

$ git commit -m 'add my README.md file'
[master (root-commit) 0105016] add my README.md file
 1 file changed, 1 insertion(+)
 create mode 100644 README.md

if we call git status again we receive a message that our local directory, the staging area and the local repo are all the same.

$ git status
On branch master
nothing to commit, working tree clean

Upload the local repo to a web repo on GitLab

GitLab is a web service for hosting Git repositories - content and version history. It has web and GUI interfaces as well as many tools for facilitating collaboration within repos.

GitLab has all the feature from gitHub, so once you learn one of them the knowledge and skills are easily transferable. GitHub servers are outside Canada, but GitLab servers are here at UBC, so you can host both private and public repos. The login to GitLab is not connected to the CWL account, so graduate students can still use GitLab even after graduating from UBC.

Branches (git checkout -b)

In many of the previous outputs we observed that we do our changes On branch master. This the main branch on git. If you need to make changes in the code, but do not want to mess up the already working code on the master branch, then you can generate a new branch, which is just a copy of the master branch and do all the changes (with tracking) on the new branch. Once you are confident the code is working you can merge the branch with the master branch.

Branches are particularly useful to:

  • test new features without breaking the current working code
  • collaborate in parallel
  • follow a risky idea that has the potential to break your working code

Before we start a new branch we have to make sure that all the all the changes in the local directory are up- to- date. Then we can create a new branch:

git checkout -b proposed-fixes master
Switched to a new branch 'proposed-fixes'

We are already in the new branch.

Now we can make changes and track them with git on the proposed-fixes branch, without affecting the working code on the master (main) branch.

We can check the git status to see if there are unstaged changes on the new branch.

git status

On branch proposed-fixes
nothing to commit, working tree clean

Now we can start making changes to the new branch. Change the README.md file by picking up a new favorite color, the use git status, add and commit to save the changes permanently to the branch proposed-fixes.

git status

On branch proposed-fixes
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git checkout -- <file>..." to discard changes in working directory)

        modified:   README.md

no changes added to commit (use "git add" and/or "git commit -a")
 git add .
git commit -m 'change favorite color in readme file'
[proposed-fixes ccdc7e9] change favorite color in readme file
 1 file changed, 2 insertions(+), 1 deletion(-)

So now we have changed the readme file on the proposed_fixes branch, but not on the master branch. to verify this:

  1. Go back to the master branch git checkout master
  2. Open the README.md file and make sure that the favorite color is back to the version before you changed it!

Merging (git merge)

Once we are confident that the code on the proposed_fixes branch is working properly, we can merge it with the master branch using git merge. To merge the proposed_fixes branch with the master branch we need to be switched on the master branch:

  1. Go back to the master branch git checkout master
  2. Merge the new branch git merge proposed_fixes which overwrites the older master branch with our new changes in the proposed_fixes.

Note: Open the README.md file to make sure that your favorite color is back to the color changed on the proposed_fixes branch.

Merging conflicts

Merging works smoothly if only the new branch being merged has been altered since it branched off. If both the new branch and the master are changed, there will be a conflict.

The conflicts of merging can be resolved manually, by editing the files and saving the new, correct merged version with add and commit.

  2. Edit to your name of choice, making sure to remove the markers.

Undoing changes (git revert)

If you change change your mind about your edits or accidentally delete a file you can undo the changes using git revert.

Let’s mimic a mistaken commit so that we can revert it back. Please make sure that everything is up-to-date on the master branch. git log --oneline will return a list of all our previous commits:

49a7a1a (HEAD -> master, origin/master) solve merging conflict
51fe9e0 remove my name from the readme file
c93d25b (proposed_changes) add my last name to the readme file
ccdc7e9 (proposed-fixes) change favorite color in readme file
6b5dee1 add my favorite color
7cf2d89 add my name to readme file
0105016 add my README.md file

Last commit is denoted by HEAD. Now let’s change our readme.md file such that we change the name or our favorite color. Then git status, add and commit git log --oneline again will show that HEAD commit is our last commit, which saves last changes to our README.md file.

57e094d (HEAD -> master) add my second last name
49a7a1a (origin/master) solve merging conflict
51fe9e0 remove my name from the readme file
c93d25b (proposed_changes) add my last name to the readme file
ccdc7e9 (proposed-fixes) change favorite color in readme file
6b5dee1 add my favorite color
7cf2d89 add my name to readme file
0105016 add my README.md file

Now we can revert back the last change, so that the HEAD will move to the reverted version. git revert HEAD --no-edit will revert the HEAD without starting the commit message editor, i.e., the commit message will be by default Revert you last commit message:

9df4431 (HEAD -> master) Revert "add my second last name"
57e094d add my second last name
49a7a1a (origin/master) solve merging conflict
51fe9e0 remove my name from the readme file
c93d25b (proposed_changes) add my last name to the readme file
ccdc7e9 (proposed-fixes) change favorite color in readme file
6b5dee1 add my favorite color
7cf2d89 add my name to readme file
0105016 add my README.md file

Note that the README.md file is back to the version before the unwanted commit. Also, note that git keeps log of the revert commit too.

Collaborate with GitLab

GitLab allows to access your repos from multiple computers. Furthermore, GitLab allows for collaboration between multiple users.

Copy an online repo (git clone)

If you are the only one working on your repo and only work on it from 1 computer, then GitLab acts as a one-way street. You push things there for storage, sharing, etc. and should only need to pull them back down if something terrible happens to your computer.

However, GitLab also allows work on a repo from multiple computers and by multiple different users. You or others can copy an online GitLab repo to your computer by cloning:

git clone URL.of.the.repo

Update from the online repo (git fetch and pull)

Now when you have multiple copies of a repo potentially being worked on and pushed to GitLab from multiple computers or people, the version on GitLab may be more up-to-date than the one you have on the computer you happen to currently be working on.

To update your local repo from GitLab, you must go through a staging area (index) similar to the one in-between your local directory and repo. You can place GitLab content in this index with

git fetch

And use git status to compare your local and GitLab versions. Then, if you decide you want to update your local version, you reverse push with

git pull

More on forking

Checking Out and Testing Pull Requests

Before you accept the Merge request, the changes on the forked branch need to be reviewed.

  • Fetch and checkout any pull request so that you can test them:
git fetch origin pull/Merge-REQUEST_NUMBER/head:pr-Merge-REQUEST_NUMBER

Note: origin might need to be replaced with upstream. Merge-REQUEST_NUMBER can be found on the GitLab interface (every merge request has its own number), pr-Merge-REQUEST_NUMBER is the name of the the local branch into which you would like the specified pull request to be fetched (you can give any name here).

  • Checkout the local branch and start reviewing the new code:
git checkout pr-Merge-REQUEST_NUMBER

Keeping the Fork Up to Date

This is useful step if you use your forked repo for long-term changes rather than one-time quick fix.

  • Add a remote upstream repo (this will be the name of the original web repo):
git remote add upstream https://gitlab.math.ubc.ca/OriginalRepo_USER/ORIGINAL-PROJECT.git
  • Verify the new remote named ‘upstream’
git remote -v
  • Whenever there is a need to update the fork with the latest upstream (original web repo) changes, fetch the upstream repo’s branches and latest commits to bring them into your repository:
git fetch upstream
  • View all branches, including those from upstream
git branch -va
  • Checkout your own master branch and merge the upstream repo’s master branch:
git checkout master
git merge upstream/master
